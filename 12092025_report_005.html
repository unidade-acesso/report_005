Requisito 1.1 - O menu de navegação deve estar estruturado como uma lista de opções

etiqueta: NOK

Lista de evidências recolhidas:

  • evidência: Uso de listas de forma incorreta

    etiqueta: chk 10 webetiqueta: NOKetiqueta: R 1.1

    O menu de navegação deve estar estruturado como uma lista de opções.

    • Ver requisito 1.1 da lista 10 aspetos

    No rodapé do site, a lista que apresenta os contactos da cm Valongo contém itens de vários tipos, sendo que alguns deles foram incorretamente autonomizados.

    Título do rodapé "Câmara Municipal de Valongo" está dentro da lista ul li

    Como se pode observar na figura, a mesma apresenta uma lista em que:

    • O primeiro item é um cabeçalho, que poderia estar fora da lista;
    • O segundo item apresenta o nome da rua e o número da porta e o terceiro item apresenta o código postal e a localidade, sendo que os URLs das hiperligações dos mesmos são iguais. Estes dois itens poderiam ser aglutinados num só, já que a sua função é apresentar um só item, que é a morada. Questões de estilo num mesmo item de lista, por exemplo, a mudança de linha que atualmente é conseguida à custa da autonomização em dois itens, pode ser conseguida no mesmo item, utilizando um elemento br;

    Recomendamos a restruturação dos itens das listas presentes no site, de modo a que cada item se baste para representar um conceito independentemente dos outros itens, e todos os itens sejam do mesmo tipo (links, texto, etc).

etiqueta: NOK

Lista de evidências recolhidas:

    etiqueta: chk 10 webetiqueta: NOKetiqueta: R 1.2

  • evidência: Não é possível navegar para a opção seguinte do menu sem percorrer as subopções

    etiqueta: chk 10 webetiqueta: melhoriaetiqueta: R 1.2

    É possível selecionar as opções e as subopções do menu quer com rato quer com teclado.

    • Ver requisito 1.2 da lista 10 aspetos

    As opções do menu não devem abrir automaticamente quando recebem o foco através das tecnologias de apoio ou do teclado. Este comportamento faz com que os utilizadores tenham que percorrer todas as subopções antes de conseguirem avançar para a próxima opção do menu. Por exemplo, quando navegamos pelo menu utilizando o teclado, não conseguimos aceder as subopções de "Participar" sem antes percorrer por todas as subopções de "Município" e "Diretório de Serviços".

    É necessário percorrer todas as subopções antes de aceder a opção "Participar" do menu

    Recomendamos que as opções de 2º nível do menu fiquem fechadas até o utilizador escolher aceder às subopções. Para isso, é necessário criar um evento no Javascript que gerencie a abertura e fecho das opções do menu pelo atributo aria-expanded.

    Para mais informações partilhamos também um exemplo de construção do menu da W3C.

  • evidência: Implementação desnecessária de técnicas de acessibilidade no menu

    etiqueta: chk 10 webetiqueta: NOKetiqueta: R 1.2

    É possível selecionar as opções e as subopções do menu quer com rato quer com teclado.

    • Ver requisito 1.2 da lista 10 aspetos

    Verificámos a utilização de técnicas de enriquecimento do menu principal com foco na acessibilidade. No entanto, identificámos que algumas dessas técnicas são redundantes e outras encontram-se incorretamente implementadas, uma vez que correspondem a menus de aplicação e não a menus de navegação, como é o caso do menu do website.

    Figura 1 - Menu principal

    Como se pode observar na figura:

    • Os elementos <ul> do menu principal foram enriquecidos com o atributo role= "menu";
    • Os elementos <li> do menu principal foram enriquecidos com o atributo role= "none";
    • Os elementos <a> dentro dos elementos <li> do menu principal foram enriquecidos com os atributos role= "menuitem", aria-haspopup="menu" e aria-expanded="false" (valor por omissão).
      Sendo este um menu de navegação (não de aplicação), a utilização da maior parte destes atributos é desnecessária.
      O único atributo com relevância neste tipo de menus é o aria-expanded presente nos elementos <a>, mas verificámos que o seu valor não é atualizado quando um item de menu é expandido ou recolhido.
      Verificámos ainda que o comportamento de navegação com as setas direcionais é inconsistente.
      Recomendamos a eliminação dos atributos role= "menu" dos elementos <ul>, role= "none" dos elementos <li> e role= "menuitem" e aria-haspopup="menu" dos elementos <a>.
      Recomendamos ainda que o comportamento dos atributos aria-expanded seja desenvolvido, de modo a que o seu valor seja atualizado ao expandir e recolher cada item de menu, passando assim essa informação às tecnologias de apoio.
      Partilhamos um exemplo de construção do menu no Codepen, com as principais características de uma menu de navegação acessível.

  • evidência: As subopções do menu não possuem um nome acessível

    etiqueta: chk 10 webetiqueta: NOKetiqueta: R 1.2
